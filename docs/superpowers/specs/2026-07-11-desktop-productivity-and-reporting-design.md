# 四象日志桌面效率与报告功能设计规格

日期：2026-07-11  
目标版本：0.3.0  
状态：待用户书面确认

## 1. 目标

在不改变现有四象日志核心布局、六套星期皮肤、任务、提醒、专注、复盘、统计、备份和中文菜单行为的前提下，增加以下能力：

1. 主界面一键循环切换皮肤。
2. Windows 桌面组件模式及仅在该模式生效的透明度调整。
3. 月历上月、下月和返回今天功能。
4. 四象限事项颜色、象限内独立编号、完成划线和独立手动划线。
5. 根据事项与复盘记录离线生成约 200 字周报和约 600 字月报。
6. 预留 AI 润色入口，但 0.3.0 不接入任何网络服务或 API Key。
7. 将项目调整为可增量开发、增量测试和快速发布的普通源码仓库结构。

## 2. 不变范围

- 保留现有主界面网格、侧栏导航和各功能入口。
- 保留周一至周六六套皮肤，周日跟随周六。
- 保留设置页的“仅今天”和“保持使用”皮肤方式。
- 保留 SQLite 本地存储、提醒、托盘、开机启动、备份恢复和单实例行为。
- 保留顶部中文应用菜单及全部中文子菜单。
- 保持业务数据默认完全离线，不增加遥测或后台网络请求。
- 原有事项和设置必须通过数据库迁移无损升级。

## 3. 主界面快捷工具条

在主界面标题右侧增加紧凑工具条。窄窗口下允许工具条换行，但不得改变主内容区的功能顺序。

### 3.1 一键换肤

- 按钮显示调色盘图标和当前皮肤短名称。
- 每次点击切换到下一套皮肤并立即写入 `themeOverride`。
- 顺序为：周一深海蓝 → 周二极光蓝 → 周三薄荷绿 → 周四治愈深绿 → 周五炭黑橙 → 周六青灰蓝 → 自动随星期。
- “自动随星期”会清除手动覆盖并立即按当天星期解析皮肤，避免用户无法回到自动模式。
- 快捷切换默认采用长期保存；“仅今天”仍在设置页中操作。

### 3.2 桌面模式

- 普通窗口显示“嵌入桌面”，桌面模式显示“恢复窗口”。
- 切换期间按钮禁用，防止连续点击产生重复窗口操作。
- 进入桌面模式前保存窗口边界、最大化状态和普通窗口透明度。
- 桌面模式隐藏系统边框和任务栏图标，保留应用内工具条与“恢复窗口”按钮。
- 普通应用窗口应覆盖桌面组件；桌面组件不得强制置顶。
- 托盘菜单始终提供“恢复主窗口”，作为窗口不可见时的恢复入口。

### 3.3 透明度

- 透明度滑块仅在桌面模式显示，范围为 40%～100%，步长 5%，默认 85%。
- 调整时即时预览，并以 150 毫秒防抖写入设置。
- 普通窗口始终恢复为 100%，不得继承桌面透明度。
- 设置值超出范围或损坏时回退到 85%。

### 3.4 报告按钮

- 工具条提供“周报”和“月报”按钮。
- 点击后打开应用内报告预览对话框，而不是直接写文件。
- 对话框提供“复制全文”“导出 TXT”“重新生成”“AI 润色”与“关闭”。
- “AI 润色”在 0.3.0 只显示“尚未配置 AI 服务，本地报告不受影响”，不得发起网络请求。

## 4. Windows 桌面组件实现

### 4.1 窗口控制边界

渲染层不得直接访问 Electron 或系统 API。新增 `WindowApi`，只暴露：

```ts
interface WindowApi {
  getDesktopState(): Promise<DesktopWindowState>;
  enterDesktopMode(): Promise<DesktopWindowState>;
  exitDesktopMode(): Promise<DesktopWindowState>;
  setDesktopOpacity(opacity: number): Promise<DesktopWindowState>;
}
```

所有调用通过 preload 和受限 IPC 完成，参数使用 Zod 验证。

### 4.2 原生桌面层

- Windows 需要将 Electron 窗口挂载到系统 `WorkerW` 桌面层，确保普通窗口能够覆盖它。
- 使用一个随应用打包的最小 C# 桌面宿主辅助程序，功能仅限：查找 `Progman/WorkerW`、挂载指定窗口句柄、解除挂载和查询父窗口。
- 辅助程序不访问网络、不读写业务数据，只接受当前应用传入的数字窗口句柄和 `attach/detach/status` 命令。
- Windows 构建流水线从仓库内的 C# 源码编译该辅助程序并打包；不得在用户机器上临时下载或编译代码。

### 4.3 失败恢复

- 找不到桌面层、辅助程序退出非零或 IPC 超时时，必须立即恢复普通窗口样式、任务栏图标、窗口位置和 100% 透明度。
- 界面显示“未能嵌入桌面，已恢复普通窗口”。
- 应用启动时若设置记录为桌面模式，但上次异常退出，则先以普通窗口显示，再由用户主动进入桌面模式。
- 单独测试托盘“恢复主窗口”、第二实例唤醒和桌面模式退出，确保不会再次出现只有后台进程而无窗口的情况。

## 5. 月份切换

- `MonthCalendar` 将“选中日期”和“当前展示月份”拆为两个状态。
- 月历标题左侧为上月按钮，右侧为下月按钮，并提供“今天”按钮。
- 上下月切换不自动创建或移动事项。
- 点击某一天才更新选中日期并加载该日事项。
- 从 1 月向前正确进入上一年 12 月，从 12 月向后正确进入下一年 1 月。
- 主工作台小月历与日历页面复用同一个月份导航组件，不保留静态写死月份。
- 日历按当前展示月份一次读取事项范围，避免逐日发起 28～31 次数据库调用。

## 6. 四象限事项显示与状态

### 6.1 象限颜色

每个象限使用稳定的语义色，同时继续适配六套皮肤：

| 象限 | 主色 | 用途 |
| --- | --- | --- |
| 重要且紧急 | 红色 | 左边框、编号底色、状态点 |
| 重要不紧急 | 蓝色 | 左边框、编号底色、状态点 |
| 紧急不重要 | 橙色 | 左边框、编号底色、状态点 |
| 不紧急不重要 | 灰绿色 | 左边框、编号底色、状态点 |

颜色不得作为唯一信息来源，象限标题与移动选择器仍保留文字。

### 6.2 编号

- 每个象限分别从 1 开始编号。
- 编号依据当前排序后的可见事项位置实时计算，不写入数据库。
- 移动、删除、恢复或重新排序后自动连续编号，不保留空号。
- 已完成但未删除的事项仍参与编号。

### 6.3 完成状态

- 每条事项前增加可操作复选框。
- 勾选调用现有完成接口，事项状态变为 `completed`、整条降低饱和度并为标题添加删除线。
- 再次点击调用恢复接口，回到 `active`。
- 完成状态继续记录 `completedAt`，供报告统计。

### 6.4 独立手动划线

- 每条事项增加“划线/取消划线”按钮，使用独立字段 `manualStruck`。
- 手动划线不修改 `status`、`completedAt`、统计或报告完成数量。
- 已完成与手动划线可以同时存在；取消手动划线不会恢复已完成事项。
- 手动划线事项仍可移动、编辑、完成、恢复和删除。

## 7. 数据库与接口

### 7.1 数据迁移

新增连续迁移，为 `tasks` 增加：

```sql
manual_struck INTEGER NOT NULL DEFAULT 0 CHECK (manual_struck IN (0, 1))
```

领域模型增加 `manualStruck: boolean`。迁移必须支持现有 0.2.0 数据库直接升级，备份恢复清单保持兼容。

### 7.2 事项与复盘范围查询

为报告和月历增加范围查询，避免循环调用每日接口：

```ts
TaskApi.listByRange(startDate: string, endDate: string): Promise<Task[]>;
ReviewApi.listByRange(startDate: string, endDate: string): Promise<DailyReview[]>;
TaskApi.setManualStruck(id: string, struck: boolean): Promise<Task>;
```

日期使用本地日历日期 `YYYY-MM-DD`，起止日期均包含。IPC 必须拒绝无效日期和起始日晚于结束日的请求。

## 8. 周报与月报

### 8.1 周期

- 周报默认统计包含当前选中日期的自然周，周一开始、周日结束。
- 月报统计当前月历展示月份的第一天至最后一天。
- 报告对话框标题明确显示起止日期。

### 8.2 本地报告内容

报告生成器为纯函数，只接收事项、复盘和日期范围，不直接访问数据库或界面。

周报目标为 180～240 个中文字符，包含：

1. 本周事项总数、完成数和完成率。
2. 重要且紧急、重要不紧急事项中的主要成果。
3. 未完成事项与风险。
4. 复盘中的收获、改进和下周重点。

月报目标为 540～660 个中文字符，增加：

1. 四象限分布与完成趋势。
2. 按周拆分的进展摘要。
3. 高频未完成事项和延期风险。
4. 下月建议与重点。

生成器优先使用真实事项标题和复盘文本；内容不足时明确写“本周期记录较少”，不得编造不存在的成果。内容过长时按段落优先级裁剪，不截断事项标题。

### 8.3 导出与 AI 预留

- 复制使用浏览器剪贴板能力。
- TXT 导出通过主进程保存对话框完成，文件名为 `四象日志-周报-YYYY-MM-DD.txt` 或 `四象日志-月报-YYYY-MM.txt`。
- AI 入口使用独立 `ReportEnhancer` 接口占位，0.3.0 实现固定的“未配置”结果。
- 不加入第三方 AI SDK、密钥字段或联网权限。

## 9. 模块划分

新增或调整的职责边界：

- `renderer/features/toolbar`：主界面快捷工具条及透明度交互。
- `renderer/features/calendar`：展示月份状态与月份导航。
- `renderer/features/tasks`：象限编号、完成和手动划线展示。
- `renderer/features/reports`：报告预览、复制、导出和本地生成。
- `main/window-control`：普通窗口与桌面模式状态机。
- `native/desktop-host`：最小 Windows 桌面层辅助程序。
- `main/repositories`：事项与复盘范围查询。
- `shared/ipc`：经过验证的窗口、报告导出和范围查询接口。

各模块不得直接读取其他模块内部状态，只通过明确的 props、领域对象或 IPC 接口通信。

## 10. 快速修改与增量验证

### 10.1 仓库结构

- GitHub 主分支直接跟踪 `src`、`tests`、`scripts`、配置和文档文件。
- 停止把源码作为 `FourQuadrantJournal-*-source.zip` 提交到仓库再由工作流解压。
- 绿色版和单文件版只作为 Release 或工作流产物，不提交大体积二进制文件到源码历史。

### 10.2 开发期验证

新增 `npm run verify:changed`：

- 根据 `git diff --name-only` 将改动映射到相关 Vitest 文件。
- 主题改动运行主题解析、设置和布局测试。
- 月历改动运行月历、日期与事项范围查询测试。
- 事项改动运行领域、仓库、四象限和统计测试。
- 窗口改动运行窗口控制、preload、托盘和启动测试。
- 报告改动运行报告生成、范围查询和导出测试。
- 公共领域类型、IPC、数据库迁移或构建配置变化时，自动升级为完整源码验证。

日常开发默认运行受影响测试和增量 Vite 构建，不生成 Windows 包。

### 10.3 持续集成与正式发布

- Pull Request 运行 `verify:changed`、类型检查和生产构建。
- 主分支普通提交运行快速验证，不打 Windows 包。
- 版本标签或手动“正式发布”才执行一次 `npm run verify`、Windows 构建、桌面模式验证和绿色版上传。
- npm、Electron 与构建工具缓存以 lockfile 哈希为键；依赖未变化时复用缓存。
- 正式发布仍必须完整验证一次，不以增量测试替代最终质量门槛。

## 11. 错误处理

- 数据库迁移失败：停止启动并提示恢复备份，不使用半迁移数据库继续运行。
- 月份范围查询失败：保留当前月份和已有事项，显示可重试提示。
- 报告生成失败：保留已生成文本，不清空对话框，允许重新生成。
- 导出取消：视为正常取消，不显示错误。
- 剪贴板权限失败：显示“复制失败，请手动选择文本”。
- 桌面嵌入失败：无条件回退普通窗口，并保留托盘恢复入口。
- 设置值损坏：皮肤回到自动模式，桌面模式关闭，透明度回到 85%。

## 12. 测试与验收

### 12.1 单元测试

- 六套皮肤与“自动随星期”的循环顺序。
- 透明度 40%～100% 限制和损坏值回退。
- 月份跨年切换、闰年和返回今天。
- 象限独立编号及移动、删除后的连续性。
- 完成状态与手动划线相互独立。
- 周报和月报周期、统计、内容缺失与字数范围。
- 改动文件到测试集合的映射。

### 12.2 集成测试

- 快捷换肤持久化及回到自动模式。
- 月历切换后一次范围查询并选择日期。
- 四象限完成、恢复、划线、取消划线与数据库持久化。
- 报告使用事项和复盘范围数据生成、复制与导出。
- 窗口 IPC 参数验证和失败回退。

### 12.3 Windows 发布验收

- 普通文件夹绿色版显示主窗口。
- 单文件绿色版显示主窗口。
- 进入桌面层后普通窗口可覆盖应用。
- 40%、85%、100% 透明度正确，普通模式恢复 100%。
- 托盘“恢复主窗口”可以从桌面模式返回。
- 应用重启后不会因上次桌面模式停留在不可见状态。
- 100%、125%、150% 缩放下工具条、月历和四象限无截断。

## 13. 完成标准

只有同时满足以下条件才可发布 0.3.0：

- 本规格列出的功能全部实现，且现有功能没有回归。
- 开发期使用增量测试；发布前完整源码验证执行一次并通过。
- Windows 文件夹版、单文件版和桌面模式真实启动验收通过。
- 最终绿色版压缩包通过完整性检查并附带中文使用说明。
- GitHub 已转换为直接跟踪源码的结构，后续小改动无需重制源码 ZIP。
